## Affected Pages/Routes
- Monitoring app backup status panel — verify the latest backup row clearly distinguishes "dump still running" vs "uploaded and verifying" vs "restore-verified"
- Monitoring app backup history list — verify older broken-format rows and new verified rows can coexist without misleading status
- Daily backup worker run — verify one scheduled run produces a remote backup object that can be downloaded and restored

## Key Interactions to Verify
- A normal daily backup completes only after the uploaded R2 file is read back and restored into the dedicated verify database
- A failed remote restore marks the backup as failed in the monitoring dashboard instead of success
- A row with `status=running`, `verification_status=pending`, and an `r2_key` is shown as "verifying", not "success"
- The backup status panel shows verification details such as format version, verification result, and verification time
- Retention still runs after a verified backup and does not hide a failed verification

## Edge Cases
- Verify DB reset leaves old rows behind before restore starts
- Backup contains JSON-heavy rows, empty tables, and sequence-backed tables
- Backup header schema fingerprint does not match the prepared verify DB
- Embedded sentinel row checksum does not match after restore
- R2 download succeeds but the gzip or SQL restore fails partway through
- Local upload succeeds but verify-time restore fails afterward
- Older backup_logs rows do not have the new verification columns populated
- SQL migration has not run yet in the target environment
- `bootstrap-redprint-schema.sql` must stay in sync with the canonical deployed schema source
- empty-DB restore in this environment needs the isolated FK-bypass fallback because normal constraint order fails on FK dependencies

## Critical Paths
- Rollout path -> `scripts/migrate-backup-logs.sql` runs first -> new worker/dashboard build deploys afterward
- Scheduled backup -> local dump creation -> R2 upload -> remote file download -> verify DB reset -> verify DB restore -> manifest verification success -> success log visible in dashboard
- Scheduled backup -> R2 upload -> remote file download -> restore failure -> failed log visible in dashboard
- Trusted bootstrap refresh -> `npm run export:bootstrap-schema` from the canonical deployed schema source -> commit `bootstrap-redprint-schema.sql` -> empty-DB recovery drill
- Empty-DB recovery rehearsal -> bootstrap schema applied to disposable Postgres -> latest v2 dump replay -> manifest + sequence smoke pass -> fallback path documented if FK order blocks the normal restore
- Backup status query -> latest entry + history query -> operator can tell whether a backup is truly restorable
