## Problem Statement

Redprint의 프로덕션 PostgreSQL DB(Railway)에 결제, 페이아웃, 크립토 정산, 유저 데이터 등 비즈니스 크리티컬 데이터가 쌓이고 있으나, 현재 백업이 전혀 없다. DB 장애, 실수로 인한 데이터 삭제, Railway 인프라 문제 발생 시 복구 수단이 없는 상태.

## Demand Evidence

- 예방 목적: 아직 데이터 손실을 경험하지 않았지만, purchases/payouts/crypto-settlement 등 금융 데이터가 존재하며 복구 불가능한 데이터 손실은 비즈니스 종료 수준의 리스크
- 현재 백업이 아무것도 없음 — Railway 기본 백업도 활용하지 않고 있음
- DB 스키마에 50+ 테이블, 금융/결제/정산 관련 테이블 다수 존재

## Status Quo

현재 워크어라운드: 없음. DB 장애 발생 시 복구 수단이 전혀 없다.

이 transcode-worker는 이미:
- PostgreSQL 연결 (Drizzle ORM + postgres.js)
- Cloudflare R2 업로드/다운로드 클라이언트
- Docker 컨테이너화 (node:20-slim + ffmpeg)
- 폴링 기반 스케줄러 (Scheduler 클래스)
- macOS 메뉴바 모니터 앱 (Tauri)

을 갖추고 있어, 백업 기능을 추가하기에 최적의 위치.

## Target User & Narrowest Wedge

**Target:** Redprint 운영자 (본인) — 프로덕션 DB 안전성 확보가 목표

**Narrowest Wedge:** 매일 1회 pg_dump → gzip → R2 업로드. 7일 리텐션. 이것만으로 "DB 날아가면 어제 상태로 복구 가능" 상태 달성.

## Constraints

1. **Railway PostgreSQL** — pg_dump를 remote로 실행해야 함 (DATABASE_URL로 접속)
2. **Docker 컨테이너** — pg_dump 바이너리를 Dockerfile에 추가 필요 (postgresql-client 패키지)
3. **R2 스토리지** — 기존 버킷 사용, `backups/` prefix로 격리
4. **리소스** — pg_dump 실행 중 CPU/메모리 사용, 트랜스코딩과 시간대 분리 권장
5. **보안** — 백업 파일에 민감 데이터 포함, R2 접근 제어 필수 (이미 private bucket)

## Premises

1. 결제/페이아웃 데이터가 있는 프로덕션 DB에 백업이 전혀 없다 — 비즈니스 연속성 리스크
2. 이 transcode-worker에 백업 기능을 추가하는 것이 자연스럽다 (DB+R2+Docker 인프라 재활용)
3. pg_dump → gzip → R2 업로드가 가장 단순한 접근
4. 백업은 트랜스코딩과 독립적으로 주기적으로 실행되어야 한다

## Approaches Considered

### Approach A: pg_dump 스크립트 + cron (Minimal Viable)

Docker 컨테이너에 pg_dump 추가, 별도 `backup.ts` 스크립트 작성. host cron 또는 launchd로 주기 실행.

- **노력:** S (human: 2시간 / CC: 15분)
- **위험:** Low
- **장점:** 가장 단순, 백업 실패시 트랜스코딩에 영향 없음
- **단점:** cron 관리 필요, 모니터링 분리됨, 리텐션 수동 관리
- **Completeness:** 7/10

### Approach B: Worker 내장 백업 스케줄러 (선택됨)

runner.ts의 main loop에 백업 스케줄러 통합. 지정된 시간(매일 새벽 4시)에 pg_dump → gzip → R2 업로드. 리텐션 정책(7일 보관, 오래된 것 자동 삭제). 모니터 앱에서 백업 상태 확인.

- **노력:** M (human: 1일 / CC: 30분)
- **위험:** Low
- **장점:** 단일 프로세스 관리, 실패 알림 통합, 리텐션 자동화, 모니터 통합
- **단점:** pg_dump 실행 중 트랜스코딩 성능 약간 영향 가능
- **Completeness:** 9/10

### Approach C: WAL 스트리밍 + PITR

Railway WAL 스트리밍 활용 지속적 복제. RPO ~0.

- **노력:** L (human: 1주 / CC: 4시간)
- **위험:** High
- **장점:** 가장 완벽한 복원
- **단점:** Railway WAL 접근 제한, 복잡도 높음, 별도 인프라 필요
- **Completeness:** 10/10 (ocean — 현재 단계에서 과도)

## Recommended Approach

**Approach B: Worker 내장 백업 스케줄러**

### 구현 상세

#### 1. Dockerfile 수정
`base` stage에 PostgreSQL 공식 apt 저장소에서 최신 클라이언트 설치. Railway PG 버전 업그레이드 시에도 pg_dump 호환성 유지. `runner` stage는 `base`를 상속하므로 `pg_dump` 바이너리 자동 사용 가능.
```dockerfile
RUN apt-get update && \
    apt-get install -y --no-install-recommends ffmpeg curl ca-certificates gnupg && \
    curl -fsSL https://www.postgresql.org/media/keys/ACCC4CF8.asc | gpg --dearmor -o /usr/share/keyrings/pgdg.gpg && \
    echo "deb [signed-by=/usr/share/keyrings/pgdg.gpg] http://apt.postgresql.org/pub/repos/apt bookworm-pgdg main" > /etc/apt/sources.list.d/pgdg.list && \
    apt-get update && \
    apt-get install -y --no-install-recommends postgresql-client-17 && \
    rm -rf /var/lib/apt/lists/*
```

#### 2. 새 파일: `src/lib/backup/backup-scheduler.ts`
- `BackupScheduler` 클래스: 매일 지정 시간(기본 04:00 UTC)에 백업 실행
- `runner.ts`의 `start()` 함수에서 main `while` loop 진입 **전에** `BackupScheduler.start()` 호출
- `stop()` 함수에서 `stopSweeper()` 옆에 `BackupScheduler.stop()` 호출
- `setTimeout` 체인 방식: 다음 백업 시간까지 남은 ms를 계산하여 한 번만 대기. 백업 완료 후 다음 날 동일 시간으로 다시 setTimeout. setInterval 1분 폴링보다 단순하고 "오늘 이미 했나?" 체크 불필요.
- **재시작 안전성:** 시작 시 R2에서 `backups/redprint-db-{today}*` 키 존재 여부 확인. 이미 존재하면 다음 날로 setTimeout.
- **graceful shutdown:** `stop()` 호출 시 진행 중인 백업이 있으면 child process 완료 대기 (타임아웃 30초). 진행 중 아니면 즉시 정리.

#### 3. 새 파일: `src/lib/backup/pg-dump.ts`
- `child_process.execFile`로 `pg_dump` 실행 — **child process**이므로 Node.js 이벤트 루프를 블로킹하지 않음. 트랜스코딩 job 폴링은 백업 중에도 정상 동작.
- plain SQL 포맷 + gzip 압축 (`.sql.gz`). `--format=custom`은 사용하지 않음 — plain SQL이 복원 시 `psql`로 직접 실행 가능하여 더 범용적.
- `pg_dump -d "$DATABASE_URL"` 형태로 연결 — URL 수동 파싱 불필요 (pg_dump가 connection URI를 네이티브 지원)
- stdout → gzip → 임시 파일로 스트리밍

#### 3.1. 무결성 검증 (CEO 리뷰 추가)
- temp 파일 크기 > 0 확인
- `gzip -t` 실행하여 gzip 무결성 검증
- 검증 실패 시: 에러 로깅 + temp 파일 삭제 + 당일 skip (손상된 파일 업로드 방지)

#### 3.2. 에러 핸들링
- 통합 try-catch: 모든 백업 단계를 감싸서 어떤 에러든 `[BACKUP]` prefix로 로깅
- `pg_dump` exit code ≠ 0: 에러 로깅 + 임시 파일 삭제 + 당일 skip
- ENOENT (pg_dump 바이너리 없음): 에러 로깅 + skip
- ENOSPC (디스크 부족): 에러 로깅 + partial temp file cleanup + skip
- R2 업로드 실패: 에러 로깅 + temp 파일 삭제 + skip
- R2 list/delete 실패: 에러 로깅 + 리텐션 skip (백업 자체는 성공)
- **CONNECTION STRING 마스킹:** 에러 로그에 DATABASE_URL 절대 출력 금지

#### 4. 새 파일: `src/lib/backup/backup-uploader.ts`
- 기존 R2 클라이언트 재활용
- key: `backups/redprint-db-{YYYY-MM-DD-HHmmss}.sql.gz`
- 업로드 후 로컬 임시파일 삭제

#### 5. 새 파일: `src/lib/backup/retention.ts`
- R2에서 `backups/` prefix로 목록 조회 (기존 `listR2Files` 활용)
- 7일 이상 된 백업 자동 삭제 (기존 `deleteMultipleFromR2` 활용)
- 최소 1개는 항상 보관 (safety net)

#### 6. `runner.ts` 수정
- `start()` 함수에서 `BackupScheduler` 인스턴스 생성 및 시작
- `stop()` 함수에서 스케줄러 정리

#### 7. 환경 변수 (`.env.worker`)
```
# DB Backup
BACKUP_ENABLED=true
BACKUP_HOUR_UTC=4          # 매일 UTC 04:00 (KST 13:00)
BACKUP_RETENTION_DAYS=7
```

#### 8. 모니터 앱 통합 (선택적 후속 작업)
- 마지막 백업 시간, 크기, 성공/실패 상태를 DB 또는 R2 메타데이터로 조회
- 모니터 앱 UI에 백업 상태 표시

### R2 저장 구조
```
backups/
├── redprint-db-2026-03-23-040000.sql.gz
├── redprint-db-2026-03-22-040000.sql.gz
├── ...
└── redprint-db-2026-03-16-040000.sql.gz  (7일 후 자동 삭제)
```

## Open Questions

1. **백업 시간대:** UTC 04:00 (KST 13:00)이 적절한가? 트래픽이 적은 시간대로 조정 필요할 수 있음
2. **백업 크기 & 비용:** 현재 DB 크기를 모름 — 첫 백업 후 R2 저장 비용 + Railway egress 비용 확인 필요. Railway는 무료 티어 이상 egress에 과금할 수 있음
3. **알림:** 백업 실패 시 알림 채널 (현재 콘솔 로그만, 향후 Slack/이메일 연동 가능)
4. **암호화:** R2는 이미 서버사이드 암호화 적용, 클라이언트 사이드 추가 암호화 필요한지
5. **복원 테스트:** 백업에서 실제 복원하는 프로세스/문서 필요

## Success Criteria

- [ ] 매일 자동으로 pg_dump → gzip → R2 업로드 실행
- [ ] 7일 리텐션 정책 자동 적용
- [ ] 백업 실패 시 콘솔 로그에 명확한 에러 출력
- [ ] Docker 재시작/재배포 후에도 스케줄 유지
- [ ] 백업 파일에서 실제 DB 복원 가능 확인 (최소 1회 수동 테스트)

## The Assignment

**첫 번째로 할 일:** 로컬에서 `pg_dump`를 수동으로 한 번 실행해서 Railway DB의 백업 크기를 확인하세요.

```bash
pg_dump -d "YOUR_DATABASE_URL" | gzip > /tmp/redprint-backup-test.sql.gz
ls -lh /tmp/redprint-backup-test.sql.gz
```

이 크기가 R2 무료 티어(10GB/월 Class A 작업)에 맞는지, 일일 백업 비용이 합리적인지 확인한 후 구현을 시작하세요. 백업이 1GB 이상이면 증분 백업이나 특정 테이블만 백업하는 전략을 재고해야 합니다.

## What I noticed about how you think

- "예방 목적"이라고 했지만, 결제/페이아웃/크립토 데이터가 있는 DB에 백업이 0개라는 건 이미 리스크가 현실화된 상태입니다. 이걸 인지하고 있다는 것 자체가 운영자 감각이 있다는 뜻입니다.

- R2를 바로 선택한 것 — 이미 가지고 있는 것을 쓰겠다는 실용적 사고.
